# Git 컨벤션

## 커밋 메시지 (Conventional Commits)

형식: `<type>(<scope>): <설명>`

- `<type>`: 변경 종류 (아래 표 참고)
- `<scope>`: 변경 범위 (선택). 예: `items`, `files`, `db`, `docs`
- `<설명>`: 한글로 명령형·현재형. 마침표 없이 간결하게.

### type 종류
| type | 용도 |
|------|------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 변경 |
| `style` | 코드 포맷팅(동작 변화 없음) |
| `refactor` | 리팩터링(기능 변화 없음) |
| `test` | 테스트 추가/수정 |
| `chore` | 빌드, 설정, 의존성 등 기타 |

### 예시
```
feat(items): 상품 CRUD 엔드포인트 추가
fix(files): CSV 빈 이름 행 검증 누락 수정
docs: README 실행 가이드 보완
test(items): 상품 삭제 404 케이스 추가
chore: pandas 의존성 추가
```

## 브랜치 네이밍

형식: `<type>/<간단한-설명>`

```
feature/item-crud
fix/csv-empty-name
refactor/db-session
docs/readme
```

- 기본 브랜치: `main`
- 작업은 항상 별도 브랜치에서 진행하고 PR로 병합한다.

## Pull Request 규칙

- **제목**: 커밋 컨벤션과 동일한 형식 사용
- **본문**에 포함할 내용:
  - 무엇을, 왜 변경했는지
  - 테스트 방법 / 확인 결과
  - 관련 이슈 번호(있으면)
- **머지 전 체크리스트**:
  - [ ] `uv run pytest` 통과
  - [ ] `uv run ruff check .` 통과
  - [ ] 새 기능에 대한 테스트 추가
  - [ ] 문서(README/CLAUDE.md) 갱신 필요 시 반영

## 푸시 / PR 승인 절차 (중요)

AI 에이전트(Claude 등)가 작업할 때 원격 반영 단계는 반드시 사용자 승인을 거친다.

- 작업이 끝나면 **로컬 커밋까지만** 진행한다.
- `git push` 및 PR 생성은 **자동으로 하지 않는다.** 사용자에게 먼저 알리고
  명시적 허락을 받은 뒤에만 진행한다.
- 승인 전에는 원격 저장소 상태(브랜치 푸시, PR 생성/변경)를 바꾸지 않는다.

## 커밋 단위 원칙

- 하나의 커밋은 하나의 논리적 변경만 담는다.
- 동작하는 상태로 커밋한다(테스트가 깨지지 않게).

## 문서 갱신 원칙 (README)

- README 는 **점진적으로 갱신**한다: 의미 있는 기능 단위가 끝나면 해당 섹션을 채운다.
- 미세 커밋(오타·리팩터링)마다 갱신하지 않는다.
- 프로젝트 말미에 전체를 한 번 다듬는다(polish).
